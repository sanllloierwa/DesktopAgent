# Desktop Agent 技术路线与实现方案

## 一、项目概述

本项目的核心目标是构建一个**桌面级 AI Agent**，能够自主完成跨平台（Web + 桌面客户端）的多步骤复杂任务。项目分为三个递进式阶段：

| 阶段 | 平台 | 核心能力要求 |
|------|------|-------------|
| 一阶段 | 知乎（Web） | 浏览器自动化 + 内容生成 + 文生图 |
| 二阶段 | WPS（客户端） | 桌面 GUI 自动化 + 文档格式操作 |
| 三阶段 | 微信（客户端） | 桌面应用操控 + 搜索与交互 |

---

## 二、核心技术路线分析

### 2.1 总体架构选型

推荐采用 **"大脑 + 双手 + 眼睛"** 三层架构：

```
┌─────────────────────────────────────────────┐
│                  大脑 (Brain)                 │
│         LLM 推理 / 任务规划 / 记忆管理          │
│         (Claude API / GPT-4 / Qwen)          │
├─────────────────────────────────────────────┤
│              双手 (Hands) - Tool Layer         │
│  ┌──────────┐  ┌──────────┐  ┌────────────┐  │
│  │ Browser  │  │ Desktop  │  │  AI Tools  │  │
│  │   Tools  │  │   Tools  │  │ (文生图等)  │  │
│  └──────────┘  └──────────┘  └────────────┘  │
├─────────────────────────────────────────────┤
│              眼睛 (Eyes) - Perception          │
│     截图分析 / DOM 解析 / 可访问性树读取         │
│     (OmniParser / GPT-4V / UIA / DOM)        │
└─────────────────────────────────────────────┘
```

**设计理念**：Agent 不直接操作像素，而是通过"观察 → 推理 → 行动 → 观察"的闭环来完成任务，具备错误恢复和自适应能力。

### 2.2 技术路线对比

| 路线 | 描述 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|---------|
| **A. 脚本驱动** | 编写固定的 Playwright/pyautogui 脚本 | 稳定可靠、执行快 | 无泛化能力、维护成本高 | 固定流程的生产环境 |
| **B. Agent 框架** | 基于 LangChain/CrewAI 构建工具调用链 | 有一定灵活性、社区生态好 | 复杂任务容易跑偏、调试困难 | 需要一定灵活性的场景 |
| **C. VLM 驱动** | 使用视觉语言模型直接看屏幕、操作界面 | 泛化能力最强、接近人类操作 | 成本高、速度慢、准确性不足 | 探索性/研究性任务 |
| **D. 混合模式（推荐）** | 结构化的工具层 + LLM 决策 + 视觉兜底 | 兼顾可靠性与灵活性 | 架构设计复杂 | **本项目最佳选择** |

**推荐路线 D**：对于登录、保存等高频固定操作使用结构化工具；对于内容创作、样式判断等需要语义理解的任务使用 LLM；视觉能力作为定位兜底和异常恢复手段。

## 三、各阶段实现方案

### 3.1 阶段一：知乎（Web 端自动化）

**技术栈**：Playwright + Claude API + 文生图 API

**流程拆解**：

```
登录 → 进入创作中心 → 写文章（LLM生成正文）→ 生成配图（文生图/Canvas/SVG）
→ 发布 → 搜索验证 → 评论互动 → 点赞/收藏/喜欢
```

**关键模块**：

| 模块 | 方案 | 备注 |
|------|------|------|
| 浏览器操控 | Playwright (Python) | 比 Selenium 更快，原生支持 async，反检测能力更强 |
| 登录态管理 | Playwright persistent context / `storageState` | 保存 cookie 和 session，避免每次登录 |
| 文章生成 | LLM API（Claude / GPT） | 根据给定主题生成结构化长文 |
| 配图生成 | 三条路径并行支持 | 见下方详述 |
| 搜索验证 | Playwright + 关键词匹配 | 发布后通过搜索标题验证文章可见性 |

**配图生成的三条技术路径**：

1. **文生图 API**：调用 DALL-E / Stable Diffusion / Midjourney API，根据文章段落生成插图
2. **HTML Canvas**：Agent 编写 JS 代码在 Canvas 上绘制图表、流程图
3. **SVG**：Agent 生成 SVG 代码，适合矢量图表、架构图

**反爬/反自动化应对**：
- 使用 Playwright 的 `stealth` 模式
- 模拟人类操作节奏（随机延迟、鼠标轨迹）
- 使用真实浏览器指纹

### 3.2 阶段二：WPS（桌面客户端自动化）

**技术栈**：pywinauto / UI Automation + COM 接口 + LLM

**流程拆解**：

```
打开 WPS → 新建文档 → 写入文章内容 → 设置标题样式 → 设置正文字体
→ 设置序号/列表 → 保存 → 导出为 PDF
```

**关键模块**：

| 模块 | 方案 | 备注 |
|------|------|------|
| 应用启动 | `subprocess` 或 `os.startfile` 打开 WPS | 或通过 COM 获取已运行的实例 |
| GUI 操控 | **首选 COM 接口**，pywinauto 兜底 | WPS 兼容微软 Office COM 接口 |
| 文字写入 | COM 接口直接操作文档对象模型 | 比模拟键盘输入更可靠 |
| 样式设置 | COM 操作 `Range.Font` / `ParagraphStyle` | 精确设置字体、字号、加粗、颜色 |
| 导出 PDF | COM 调用 `ExportAsFixedFormat` | 原生导出保真度高 |
| 视觉定位 | UIA / 截图 + VLM 辅助 | 用于非标准弹窗或异常情况 |

**为什么首选 COM 而非纯 GUI 模拟**：

```
GUI 模拟（pyautogui / pywinauto 点击菜单）
  ✗ 依赖屏幕分辨率、窗口位置
  ✗ 容易受弹窗、焦点变化影响
  ✗ 速度慢

COM 接口操作
  ✓ 直接操作文档对象，不依赖 UI 状态
  ✓ 快速、精确、可后台运行
  ✓ WPS 高度兼容 Office COM
```

**兜底方案**：当 COM 接口不可用时（如某些 WPS 版本差异），回退到 pywinauto + UI Automation 的 GUI 操作路径。

### 3.3 阶段三：微信（桌面客户端自动化）

**技术栈**：pywinauto + UI Automation + 图像识别

**流程拆解**：

```
打开微信 → 点击搜索 → 输入"火眼审阅" → 筛选服务号 → 点击关注 → 发送私信
```

**关键挑战与方案**：

| 挑战 | 方案 |
|------|------|
| 微信客户端无 API | 纯 GUI 自动化，通过 UI Automation Tree 定位控件 |
| 搜索结果筛选（区分服务号/联系人/文章） | 利用 UIA 控件类型和名称属性筛选，视觉识别兜底 |
| 关注按钮状态判断 | 先检测按钮文本（"关注"/"进入公众号"），避免重复操作 |
| 私信输入 | 定位聊天输入框 UIA 元素，模拟输入 |

**微信自动化特有问题**：
- 微信 PC 端是 Electron + 原生混合架构，部分控件在 UIA 树中不可见
- 可能出现定制渲染的控件，需要图像识别兜底
- 微信安全策略可能检测异常操作，需要拟人化操作节奏

## 四、Agent 核心引擎设计

### 4.1 Agent 循环

```
      ┌──────────────────────────────────┐
      │           任务输入                 │
      └──────────────┬───────────────────┘
                     ▼
      ┌──────────────────────────────────┐
      │      Planner（规划器）             │
      │  将高层目标分解为原子操作序列        │
      └──────────────┬───────────────────┘
                     ▼
      ┌──────────────────────────────────┐
  ┌──►│    Executor（执行器）              │
  │   │  调用对应 Tool 执行当前步骤         │
  │   └──────────────┬───────────────────┘
  │                  ▼
  │   ┌──────────────────────────────────┐
  │   │    Observer（观察器）              │
  │   │  获取反馈：截图/DOM/返回值          │
  │   └──────────────┬───────────────────┘
  │                  ▼
  │   ┌──────────────────────────────────┐
  │   │    Verifier（验证器）              │
  │   │  判断步骤是否成功                  │
  │   └──────────────┬───────────────────┘
  │                  │
  │        ┌─────────┴─────────┐
  │        ▼                   ▼
  │   ┌─────────┐        ┌─────────┐
  │   │  成功    │        │  失败    │
  │   └────┬────┘        └────┬────┘
  │        │                  ▼
  │        │        ┌──────────────────┐
  │        │        │  Replanner       │
  │        │        │  分析失败原因     │
  │        │        │  调整执行策略     │
  │        │        └────────┬─────────┘
  │        │                 │
  │        ▼                 ▼
  │   ┌──────────────────────────────────┐
  └───┤    下一步 / 任务完成              │
      └──────────────────────────────────┘
```

### 4.2 Tool 定义（Agent 可调用的工具集）

```python
# 浏览器工具
browser_navigate(url)           # 导航到URL
browser_click(selector)         # 点击元素
browser_type(selector, text)    # 输入文本
browser_screenshot()            # 截图当前页面
browser_get_dom()               # 获取DOM摘要
browser_search(keyword)         # 搜索

# 桌面工具
desktop_launch(app_name)        # 启动应用
desktop_find_element(desc)      # 查找UI元素
desktop_click(x, y)             # 点击坐标
desktop_type(text)              # 键盘输入
desktop_screenshot()            # 桌面截图
desktop_get_uia_tree()          # 获取UI Automation树

# AI 工具
ai_generate_article(topic)      # LLM生成文章
ai_generate_image(prompt)       # 文生图
ai_generate_svg(description)    # 生成SVG图表
ai_analyze_screenshot(img)      # VLM分析截图
ai_verify_result(expected, actual) # 验证任务结果
```

### 4.3 记忆系统

| 记忆类型 | 作用 | 实现 |
|----------|------|------|
| **短期记忆** | 当前任务上下文、已完成的步骤 | 对话历史 / 滑动窗口 |
| **工作记忆** | 当前会话状态（登录态、窗口句柄等） | 结构化 State 对象 |
| **长期记忆** | 历史成功经验、常见错误恢复策略 | 向量数据库（ChromaDB / LanceDB） |
| **空间记忆** | UI 元素位置、常用控件坐标 | 缓存 UIA 路径、CSS 选择器 |

## 五、扩展方向

### 5.1 短期扩展（项目内可完成）

1. **多平台文章发布**
   - 从知乎扩展到：CSDN、掘金、SegmentFault、微信公众号
   - 一次创作，多平台分发

2. **文章质量增强**
   - 集成 SEO 优化建议
   - 自动生成摘要、目录
   - 多语言翻译与发布

3. **智能配图引擎**
   - 根据文章语义自动选择配图位置
   - 数据可视化（表格自动转图表）
   - 代码块语法高亮截图

4. **任务录制与回放**
   - 记录用户操作轨迹，回放执行
   - 支持参数化（如更换文章主题）

### 5.2 中期扩展

5. **Agent 对话式交互**
   - 自然语言下达任务："帮我在知乎发一篇关于 Python 协程的文章"
   - Agent 主动提问澄清模糊需求
   - 执行过程实时反馈，支持中断和调整

6. **多 Agent 协作**
   - 研究 Agent：搜集资料、整理大纲
   - 写作 Agent：根据大纲生成文章
   - 配图 Agent：同步生成插图
   - 发布 Agent：执行发布和互动

7. **定时与触发式任务**
   - 定时发布（如每周一上午 10 点）
   - 事件触发（如热点新闻出现时自动写稿）
   - 条件触发（如粉丝数达标时执行特定操作）

8. **跨应用工作流**
   - 从飞书/钉钉获取任务 → WPS 生成报告 → 邮件发送
   - 浏览器抓取数据 → Excel 整理分析 → PPT 生成汇报

### 5.3 长期扩展

9. **通用桌面 Agent 平台**
   - 插件化架构：新应用通过配置文件注册即可操控
   - 社区共建的应用适配库
   - 可视化工作流编排（拖拽式 DAG）

10. **本地化与隐私**
    - 完全本地运行的 LLM（Ollama / vLLM + Qwen/Llama）
    - 本地文生图（Stable Diffusion + ComfyUI）
    - 所有数据不出本机，适合企业保密场景

11. **RPA 能力接入**
    - 对接 UiPath / Power Automate 生态
    - 将 AI Agent 的决策能力与传统 RPA 的执行可靠性结合

12. **企业级功能**
    - 操作审计日志
    - 多人协作与权限管理
    - 任务队列与优先级调度
    - 执行结果日报/周报自动生成

## 六、技术风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 知乎反爬升级 | 阶段一受阻 | 多账号轮换、降低操作频率、备用 playwright-stealth |
| WPS 版本 COM 接口差异 | 阶段二某些操作失败 | 建立多版本适配层，GUI 操作兜底 |
| 微信客户端更新导致 UIA 变化 | 阶段三定位失败 | 多策略定位（Name + ClassName + 图像匹配）|
| LLM 输出不稳定 | 文章质量波动 | 设置结构化 Prompt、输出校验、人工审核开关 |
| 文生图 API 限流/费用 | 配图成本过高 | 优先使用 Canvas/SVG 自绘，文生图作备选 |
| 操作系统兼容性 | 仅限 Windows | 初期聚焦 Windows，后续用跨平台方案扩展 |

## 七、推荐技术栈总结

```
语言：        Python 3.11+
Agent 框架：   LangChain / 自研 Agent Loop
浏览器自动化： Playwright
桌面自动化：   pywinauto + UI Automation Wrapper + COM (win32com)
LLM：         Claude API / GPT-4o（决策 + 内容生成）
VLM：         GPT-4V / Claude Vision（屏幕理解）
文生图：       DALL-E 3 / Stable Diffusion API
向量数据库：   ChromaDB（记忆系统）
包管理：       Poetry / uv
日志：        loguru
配置：        YAML + Pydantic Settings
测试：        pytest + Playwright fixtures
```

## 八、开发里程碑建议

| 里程碑 | 内容 | 预计产出 |
|--------|------|---------|
| M1 | Agent 核心框架搭建 | Agent 循环、Tool 注册机制、基础浏览器/桌面工具 |
| M2 | 阶段一 MVP | 知乎自动登录→发文章→搜索→互动全流程跑通 |
| M3 | 阶段二 MVP | WPS 文档创建→格式化→导出 PDF 全流程 |
| M4 | 阶段三 MVP | 微信搜索→关注→私信全流程 |

| M6 | 扩展与打磨 | 配图增强、UI 交互界面、文档与演示 |

---

> **核心设计原则**：优先结构化工具（可靠），LLM 做语义决策（智能），视觉做兜底（鲁棒）。不打无准备之仗——每一步操作前先观察，每一步操作后做验证。
