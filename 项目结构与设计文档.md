## 二、模块职责详解

### 2.1 `src/agent/` — Agent 核心引擎

这是项目的大脑，负责决策与调度。

```
┌──────────────────────────────────────────────────┐
│                    agent/loop.py                  │
│                                                   │
│   while not task_completed:                       │
│       context = observer.gather()                 │
│       step = planner.plan(task, context)          │
│       result = executor.run(step)                 │
│       ok, reason = verifier.check(step, result)   │
│       if not ok:                                  │
│           context.errors.append(reason)           │
│           planner.replan(context)                 │
│       memory.commit(step, result)                 │
└──────────────────────────────────────────────────┘
```

| 模块 | 职责 | 核心接口 |
|------|------|---------|
| `planner.py` | 将高层自然语言目标分解为有序工具调用序列 | `plan(task, context) → List[Step]` |
| `executor.py` | 查找对应 Tool 实例，传入参数并执行 | `run(step) → ActionResult` |
| `observer.py` | 聚合多种感知源（截图、DOM、UIA、日志） | `gather() → Context` |
| `verifier.py` | 根据预期效果判断步骤是否成功 | `check(step, result) → (bool, reason)` |
| `memory.py` | 管理三类记忆的读写与检索 | `commit() / recall() / forget()` |
| `state.py` | 持有当前会话的可变状态（窗口句柄、登录态等） | `AgentState` dataclass |

### 2.2 `src/tools/` — 工具层

每个工具继承 `BaseTool`，注册到 `ToolRegistry`，Agent 通过 function calling 或 ReAct 模式调用。

```python
# tools/base.py 核心设计
from abc import ABC, abstractmethod
from pydantic import BaseModel
from typing import Any

class ToolSchema(BaseModel):
    """工具的 OpenAI function-calling 兼容描述"""
    name: str
    description: str
    parameters: dict  # JSON Schema

class BaseTool(ABC):
    schema: ToolSchema

    @abstractmethod
    async def execute(self, **kwargs) -> Any:
        """执行工具，返回结构化结果"""
        ...

    @abstractmethod
    def validate(self, **kwargs) -> bool:
        """参数校验"""
        ...

class ToolRegistry:
    """工具注册中心，支持按名称查找"""
    def register(self, tool: BaseTool): ...
    def get(self, name: str) -> BaseTool: ...
    def list_schemas(self) -> list[ToolSchema]: ...
```

**浏览器工具子模块**（`tools/browser/`）：

| 工具 | 描述 | 关键参数 |
|------|------|---------|
| `navigate` | 导航到指定 URL | `url`, `wait_until` |
| `click` | 点击页面元素 | `selector`, `timeout` |
| `type_text` | 在输入框输入文本 | `selector`, `text`, `delay` |
| `scroll` | 滚动页面 | `direction`, `amount` |
| `screenshot` | 页面截图 | `full_page`, `element_selector` |
| `get_dom` | 获取 DOM 摘要（给 LLM 看的精简版） | `depth`, `filter_interactive` |
| `extract_text` | 提取可见文本 | `selector` |

**桌面工具子模块**（`tools/desktop/`）：

| 工具 | 描述 | 关键参数 |
|------|------|---------|
| `launch_app` | 启动桌面应用 | `app_name`, `wait_time` |
| `find_element` | 按描述查找 UI 元素 | `name`, `control_type`, `class_name` |
| `click_element` | 点击元素 | `element_id` |
| `type_text` | 键盘输入 | `text`, `interval` |
| `press_keys` | 组合键 | `keys`（如 `"Ctrl+S"`) |
| `get_uia_tree` | 获取 UI Automation 树摘要 | `depth`, `scope` |

**AI 工具子模块**（`tools/ai/`）：

|------|------|---------|
| `generate_article` | LLM 生成长文 | `topic`, `style`, `length`, `outline` |
| `generate_image` | 文生图 | `prompt`, `size`, `style` |
| `generate_chart` | 生成 SVG/Canvas 图表 | `chart_type`, `data`, `description` |
| `analyze_screen` | VLM 理解屏幕内容 | `image_base64`, `question` |
| `summarize` | 文本摘要 | `text`, `max_length` |

### 2.3 `src/perception/` — 感知层

感知层提供标准化的环境信息输入，屏蔽底层技术细节。

```python
# 统一的感知结果
@dataclass
class PerceptionResult:
    source: str           # "dom" | "screenshot" | "uia_tree" | "com"
    summary: str          # 人类可读的文本描述（给 LLM 看）
    raw: Any              # 原始数据（程序使用）
    elements: list[UIElement]  # 可交互元素列表
    timestamp: float
```

| 模块 | 输入 | 输出 | 核心技术 |
|------|------|------|---------|
| `dom_parser.py` | Playwright Page | 精简 DOM 摘要（去噪，只保留可交互元素） | CSS selector 过滤 + 可访问性属性提取 |
| `uia_parser.py` | UIA 树 | 可交互控件列表（去噪，扁平化） | `uiautomation` 库 + 自定义过滤器 |
| `screenshot.py` | 屏幕/窗口 | base64 编码的截图 | `PIL` + `mss`（高性能截图）|
| `image_matcher.py` | 模板图 + 目标区域 | 匹配坐标 + 置信度 | OpenCV 模板匹配 / 特征点匹配 |

### 2.4 `src/platforms/` — 平台流程层

每个平台文件封装一个完整的任务流程，是 **编排代码**，不是工具代码。

```python
# platforms/zhihu.py 示意
class ZhihuPlatform:
    def __init__(self, agent: AgentLoop):
        self.agent = agent

    async def publish_article(self, topic: str):
        """知乎发文章全流程"""
        # 将高层次目标交给 Agent 执行
        task = Task(
            goal=f"在知乎发布一篇关于'{topic}'的文章",
            subtasks=[
                "确保已登录知乎",
                "进入创作中心写文章",
                f"正文围绕'{topic}'展开，1500字左右",
                "为文章配2-3张相关图片",
                "发布文章",
                "通过搜索验证文章已发布",
                "给文章点喜欢和收藏"
            ]
        )
        return await self.agent.run(task)
```

**设计原则**：platforms 层只做编排，不直接调工具——所有工具调用都通过 Agent 引擎中转，这样 Agent 能获得完整的执行上下文和错误恢复能力。

## 三、数据流设计

### 3.1 任务生命周期

```
Task(goal, params)
  │
  ▼
Planner ─── 生成 ──→ [Step, Step, Step, ...]
                        │
                        ▼
                   Executor ─── 查找 ──→ ToolRegistry.get(tool_name)
                        │                     │
                        │                     ▼
                        │              Tool.execute(params)
                        │                     │
                        ▼                     ▼
                   Observer ◀─────── ActionResult(raw, summary, screenshot)
                        │
                        ▼
                   Verifier ─── 判断 ──→ (True/False, reason)
                        │
              ┌─────────┴──────────┐
              ▼                    ▼
          成功: 下一步          失败: Replan → 新 Step
              │                    │
              └────────┬───────────┘
                       ▼
              Memory.commit(step, result)
```

### 3.2 核心数据类型

```python
# types/task.py

@dataclass
class Step:
    id: str
    tool_name: str
    params: dict[str, Any]
    description: str          # 这一步在做什么（给人看）
    expected_outcome: str     # 预期结果描述（给 Verifier 看）
    retry_policy: RetryPolicy

@dataclass
class ActionResult:
    step_id: str
    success: bool
    data: Any                 # 工具返回值
    summary: str              # 人类可读的执行摘要
    screenshot_base64: str | None
    error: str | None
    duration_ms: int

@dataclass
class Task:
    id: str
    goal: str                 # 自然语言目标
    context: dict             # 额外上下文
    steps: list[Step]         # 由 Planner 填充
    status: TaskStatus        # pending | running | success | failed
    created_at: float
    completed_at: float | None
```

## 六、关键设计决策

### 6.1 为什么工具层按"能力域"而非"平台"划分？

```
✗ 按平台划分（不推荐）:           ✓ 按能力域划分（推荐）:
  tools/zhihu/                      tools/browser/
    ├── login.py                      ├── navigate.py
    ├── write_article.py              ├── interact.py
    └── ...                           └── ...
  tools/wps/                        tools/desktop/
    ├── new_doc.py                    ├── app_control.py
    └── ...                           └── ...
```

**原因**：浏览器工具（点击、输入、截图）对知乎、CSDN、掘金都通用；桌面工具（窗口管理、UIA 控制）对 WPS、微信、记事本都通用。按能力域划分避免重复，新增平台只需组合已有工具。

### 6.2 为什么 platforms 和 tools 分离？

- `tools/`：**原子能力**——打开浏览器、点击按钮、输入文字。不知道业务上下文。
- `platforms/`：**业务流程**——知乎发文章的完整流程编排。知道"先登录再写文章再配图"的顺序。

分离后：
- 工具可在不同平台间复用
- 流程变更不影响工具实现
- 测试时可分别 mock

### 6.3 COM 接口 vs 纯 GUI 模拟的策略

```
优先级: COM API → UI Automation → 图像识别

WPS:     COM（主力）  → UIA（部分操作） → 图像（弹窗关闭）
微信:    UIA（主力）  → 图像（定制控件）→ 键盘（快捷键）
```

COM 接口直接操作文档对象模型，速度快、不依赖屏幕，但 WPS 不同版本兼容性需要适配。UIA 是 Windows 标准可访问性接口，微信的 Electron 部分支持良好，但定制渲染的控件回落到图像识别。

### 6.4 登录态管理策略

| 平台 | 方案 |
|------|------|
| 知乎 | Playwright `storageState` 持久化 cookie + localStorage |
| WPS | 无登录态问题（本地应用） |
| 微信 | `pywinauto` 维持窗口存活，首次手动扫码后保持登录 |

## 七、测试设计哲学：验证泛化能力，而非脚本回放

### 7.1 核心原则

传统自动化测试的思路是"固定输入 → 固定输出"：给 Agent 一个写死的任务，断言它产出了预期的结果。这种方式在普通软件中有效，但对 AI Agent 来说完全走错了方向——它测试的是"这个脚本能不能跑通"，而不是"Agent 是否真的具备了理解需求并组合工具的能力"。

**正确的测试思路**：

```
✗ 错误方式：固定测试样例                        ✓ 正确方式：验证泛化能力
  "帮我发布一篇关于 Python 的文章"                  随机生成任务 → Agent 动态规划 → 验证工具调用链
  断言：文章标题 = "Python入门"                    不验证具体内容，验证决策路径是否正确
```

测试的核心问题是：**当任务不在任何示例中时，Agent 是否仍然能根据需求灵活地选择和组合工具来完成任务？**

### 7.2 测试分层

```
┌─────────────────────────────────────────────────────────┐
│              第三层：泛化能力测试（test_capability/）         │
│  随机生成多样化任务 → Agent 自行规划 → 验证工具调用链合理      │
│  目标：回答"Agent 能不能处理没见过的任务？"                  │
├─────────────────────────────────────────────────────────┤
│              第二层：工具单元测试（test_tools/）             │
│  每个工具的输入输出正确性，mock 外部依赖                     │
│  目标：回答"工具本身有没有 bug？"                          │
├─────────────────────────────────────────────────────────┤
│              第一层：Agent 核心测试（test_agent/）           │
│  Planner / Verifier / Memory 等模块的逻辑正确性             │
│  目标：回答"引擎各零件是否正常工作？"                        │
└─────────────────────────────────────────────────────────┘
```

### 7.3 泛化能力测试设计（核心）

不是写死一个任务让 Agent 执行，而是：

**Step 1 — 随机生成任务描述**：`task_generator.py` 从工具和能力空间中随机组合出新任务，确保测试覆盖已知示例之外的场景。

```python
# harness/task_generator.py 示意
class TaskGenerator:
    """随机生成任务，不依赖固定模板"""

    platforms = ["知乎", "CSDN", "掘金", "微信公众号", "Notion", "飞书文档"]
    actions = ["发布文章", "搜索内容", "评论", "点赞", "导出PDF", "格式化文档", "发送私信"]
    constraints = ["包含配图", "字数1500+", "标题加粗", "添加序号列表", "嵌入代码块"]

    def generate(self, seed: int) -> Task:
        """每次生成不同的任务组合"""
        rng = random.Random(seed)
        platform = rng.choice(self.platforms)
        num_actions = rng.randint(2, 5)
        selected_actions = rng.sample(self.actions, num_actions)
        selected_constraints = rng.sample(self.constraints, k=rng.randint(1, 3))
        return Task(
            goal=f"在{platform}上{'并'.join(selected_actions)}，要求{'，'.join(selected_constraints)}",
            ...
        )
```

**Step 2 — Agent 自主规划**：将自然语言任务交给 Planner，Agent 自己决定要用哪些工具、按什么顺序执行，不提供任何预设步骤。

**Step 3 — 验证决策路径**：不是验证"文章标题是否等于某个字符串"，而是验证 Agent 的工具调用链是否合理：

```python
# harness/metrics.py 示意
class GeneralizationMetrics:
    """评估 Agent 泛化能力的指标，不评估具体输出内容"""

    def evaluate(self, task: Task, trace: ToolTrace) -> dict:
        return {
            # Agent 的规划是否包含了完成此任务必需的工具类型？
            "tool_coverage": self._check_required_tools(task, trace),
            # 工具调用顺序是否符合逻辑约束？（如"发布"必须在"写文章"之后）
            "ordering_valid": self._check_ordering(trace),
            # 遇到失败时是否尝试了替代路径？
            "recovery_attempted": self._count_recovery_paths(trace),
            # 是否有冗余的工具调用（来回跳转、重复操作）？
            "efficiency": self._measure_efficiency(trace),
            # 规划的参数是否从任务描述中正确推导（而非硬编码）？
            "parameter_derivation": self._check_params_derived(task, trace),
        }
```

### 7.5 两类测试对比

| 维度 | 固定样例测试（不推荐） | 泛化能力测试（推荐） |
| --- | --- | --- |
| 任务来源 | 人工提前写死 | 随机组合生成，每次不同 |
| 验证内容 | 输出内容是否匹配预期值 | 工具调用链是否合理 |
| 覆盖范围 | 只覆盖已写好的几个场景 | 覆盖工具组合空间 |
| 维护成本 | 每加一个平台都要写新测试 | 工具不变，测试不用改 |
| 回答的问题 | "这个脚本还能跑吗？" | "Agent 能应对新需求吗？" |
| 对真实环境依赖 | 强依赖，网络/平台一变就挂 | 通过 mock 隔离 |

### 7.6 示例：一个泛化能力测试

```python
# tests/test_capability/test_web_tasks.py
import pytest
from harness.task_generator import TaskGenerator
from harness.env_simulator import EnvironmentSimulator
from harness.metrics import GeneralizationMetrics

@pytest.mark.parametrize("seed", range(20))  # 跑 20 个随机任务
async def test_agent_handles_arbitrary_web_task(seed, agent, simulator):
    """验证：对于任意 Web 类任务，Agent 都能规划出合理的工具调用链"""
    task = TaskGenerator(platforms=["知乎", "CSDN", "掘金"]).generate(seed)
    simulator.setup_browser(scenario=task.goal)

    trace = await agent.run(task)
    scores = GeneralizationMetrics().evaluate(task, trace)

    assert scores["tool_coverage"] >= 0.8, \
        f"工具覆盖不足: 任务需要但 Agent 没调用的工具 {scores['missing_tools']}"
    assert scores["ordering_valid"], \
        f"工具调用顺序不合法: {scores['ordering_errors']}"
    assert scores["parameter_derivation"] >= 0.7, \
        f"参数未从任务描述中推导: {scores['hardcoded_params']}"
```

关键点：测试中对每个 seed 生成的是**不同的任务**，Agent 必须根据任务描述**动态决定**用什么工具。20 个 seed 覆盖 20 种不同的任务组合，远超手写 3 个固定场景的覆盖面。

### 7.7 异常恢复测试

专门验证 Agent 在工具失败时能否找到替代路径——这也是泛化能力的一部分：

```python
# tests/test_capability/test_error_recovery.py
async def test_agent_recovers_from_element_not_found(agent, simulator):
    simulator.setup_browser(scenario="写文章")
    simulator.inject_anomaly("publish_button_hidden")  # 发布按钮找不到

    trace = await agent.run(Task(goal="发布文章"))

    # Agent 应该尝试其他方式：快捷键 Ctrl+Enter、滚动页面找按钮、检查是否已登录等
    assert trace.has_alternative_path(), "Agent 没有尝试替代方案就放弃了"
    assert not trace.is_loop(), "Agent 陷入了死循环（反复尝试同一操作）"
```
