# Desktop Agent 项目分析报告

> 分析日期：2026-06-23

---

## 一、项目概述

Desktop Agent 是一个跨平台 AI 桌面自动化代理，核心理念是 **"Brain + Hands + Eyes"** 三层架构：

- **Brain（大脑）**: LLM 驱动的规划、推理与记忆系统
- **Hands（双手）**: 18 个原子工具，覆盖桌面（WPS COM）、浏览器（Playwright）、AI 能力
- **Eyes（眼睛）**: 截图感知（mss+PIL）+ Windows UI Automation 树解析

Agent 执行模型为 **Plan → Execute → Observe → Verify** 闭环，支持失败重试和重新规划。

---

## 二、架构与设计逻辑

### 2.1 核心数据流

```
用户输入(自然语言)
    │
    ▼
Planner (LLM 将目标分解为 Step 序列)
    │
    ▼
AgentLoop ──→ Executor ──→ Tool.execute()
    │            │
    │            ▼
    │         Observer (聚合 DOM/UIA/截图)
    │            │
    │            ▼
    │         Verifier (规则 + LLM 视觉验证)
    │            │
    │       ┌────┴────┐
    │    成功         失败
    │     │            │
    │     ▼            ▼
    │  下一步     重试/Replan
    │
    ▼
Memory (三层记忆：短期/工作/长期)
    │
    ▼
EventBus ──→ UI (Gradio / CustomTkinter)
```

### 2.2 三层记忆系统

| 层级 | 实现 | 作用 |
|------|------|------|
| 短期记忆 | `deque`（50条滑动窗口） | 当前任务的最近步骤和观察 |
| 工作记忆 | `WorkingMemory` 实例 | 当前会话的完成步骤、错误列表、草稿区 |
| 长期记忆 | ChromaDB 向量数据库 | 跨任务的模式记忆和成功经验 |

### 2.3 LLM 适配器模式

```
create_llm_client()
    ├── provider="anthropic"  → _AnthropicAdapter  (原生 SDK)
    ├── provider="deepseek"   → _DeepSeekAdapter   (OpenAI 兼容)
    ├── provider="openai"     → _DeepSeekAdapter   (同名复用)
    └── provider="local"      → _DeepSeekAdapter   (Ollama 兼容)
```

统一接口：所有适配器都提供 `.model` 和 `.messages.create()` 方法，返回值都支持 `.content[0].text` 访问模式。

### 2.4 事件驱动 UI 架构

```
AgentLoop (主线程/后台线程)
    │  emit(AgentEvent)
    ▼
EventBus
    │  subscribe(handler)
    ▼
UIBridge / AgentThread (后台线程)
    │  queue.put(event_dict)
    ▼
Gradio Generator / CTK Polling (UI 线程)
    │  yield HTML / after(300ms)
    ▼
用户界面更新
```

### 2.5 工具注册模式

所有工具通过 `register_all_tools()` 一次性注册到 `ToolRegistry`，每个工具继承 `BaseTool`，必须定义：
- `schema: ToolSchema`（名称、描述、参数 JSON Schema）
- `async def execute(**kwargs) -> dict`（执行逻辑）

## 三、设计优点

1. **COM 优先策略**：WPS/Word 操作优先使用 COM 接口（快速、可靠、可后台运行），pywinauto/UIA 作为补充
2. **适配器模式**：不同的 LLM provider 共享统一接口，切换 provider 只需修改配置
3. **事件解耦**：Agent 核心与 UI 层通过 EventBus 完全解耦，Gradio 和 CustomTkinter 可互换
4. **安全执行**：`BaseTool.safe_execute()` 统一捕获异常，保证单个工具失败不会导致整个 Agent 崩溃
5. **配置分层**：default.yaml → 覆盖文件 → 环境变量 → UI 设置，优先级清晰
6. **双验证机制**：规则验证（快速）+ LLM 视觉/文本验证（深度），平衡速度和准确性

## 四、可能导致项目无法运行的问题

### 4.1 致命问题（必须修复）

#### A. COM 多线程初始化缺失

**文件**: [src/tools/desktop/wps_com.py](src/tools/desktop/wps_com.py)

`_get_word_app()` 函数在任何调用线程中直接使用 `win32com.client`，但没有调用 `pythoncom.CoInitialize()`。当从 Gradio 后台线程或 CustomTkinter 的 AgentThread 中调用 WPS 工具时，会出现：

```
pywintypes.com_error: (-2147221008, '尚未调用 CoInitialize。', None, None)
```

**影响范围**: 所有 WPS/Word 桌面工具在 UI 模式下全部不可用。

**修复建议**: 在 `_get_word_app()` 开头调用 `pythoncom.CoInitialize()`（需 `import pythoncom`）。

#### B. Playwright 浏览器未安装

**文件**: [src/tools/browser/navigate.py](src/tools/browser/navigate.py)

`pip install playwright` 只安装 Python 包，还需要运行 `playwright install chromium` 来下载 Chromium 浏览器。如果跳过这一步，所有浏览器工具（navigate、click、type_text 等）都会失败：

```
playwright._impl._errors.Error: Executable doesn't exist at ...
```

**修复建议**: 在 `_get_page()` 中捕获此错误并提供清晰的提示信息，或项目提供安装脚本。

#### C. API Key 未配置时程序静默失败

**文件**: [src/utils/secret.py](src/utils/secret.py)

当没有配置 API Key 时，`get_api_key()` 抛出 `ValueError`。在 CLI 模式下这会导致程序退出并显示错误，但在 UI 模式下（`UIBridge.run()`），异常被捕获并放入事件队列，UI 显示一条错误信息后进入永久等待状态，用户界面不会给出明确的下一步指导。

**修复建议**: UI 模式下在启动时主动检查 Key 配置状态，引导用户到设置页。

### 4.2 功能缺失问题

#### D. 微信操作工具不存在

**文件**: [src/platforms/wechat.py](src/platforms/wechat.py)

`--wechat` 模式构建了任务目标（"在微信中搜索、关注、发送私信"），但 `registry_init.py` 中**没有注册任何微信专用工具**。Planner 会尝试用现有的 18 个工具来完成微信任务，但它们都无法操作微信客户端，导致所有微信任务必然失败。

目前无 `wechat_search`、`wechat_follow`、`wechat_send_message` 等工具。

#### E. 知乎平台工具不存在

**文件**: [src/platforms/zhihu.py](src/platforms/zhihu.py)

类似微信，`--zhihu` 模式没有知乎专用工具（如 `zhihu_login`、`zhihu_publish_article`）。浏览器工具可以操作网页，但缺少登录态管理、知乎编辑器特定交互逻辑。

#### F. 感知层与 Agent 循环脱节

**文件**: [src/agent/observer.py](src/agent/observer.py) + [src/agent/loop.py](src/agent/loop.py)

`Observer.gather()` 只从 `AgentState` 读取缓存的感知数据，但**没有任何工具在执行后主动调用 `observer.feed_perception()`** 来更新这些缓存。意味着：

- `Observer.gather()` 永远返回空 Context
- `Verifier._llm_verify()` 获得的 `ctx.to_text()` 只显示 `"(no context available)"`
- LLM 视觉验证形同虚设

### 4.3 设计与实现不一致

#### G. Verifier 引用了不存在的工具名

**文件**: [src/agent/verifier.py:57](src/agent/verifier.py#L57)

```python
visual_tools = {"click", "navigate", "type_text", "click_element", "launch_app"}
```

`click_element` 工具不存在，在 18 个注册工具中找不到。这是一个死代码路径。

#### H. WPS/WeChat 快捷模式丢失上下文信息

**文件**: [src/main.py:100-110](src/main.py#L100-L110)

```python
task = build_wps_task(args.wps)
asyncio.run(_run_with_tools(task.goal, args.config))  # 只传 goal，丢弃了 context
```

`build_wps_task()` 返回的 Task 包含 `context={"platform": "wps", "topic": topic}`，但 `_run_with_tools` 用 `task.goal` 重新构建了 Task，**context 信息完全丢失**。

#### I. humanize 和 retry 模块未被使用

- [src/utils/humanize.py](src/utils/humanize.py) — `random_delay`、`human_typing_interval`、`generate_mouse_path` 三个函数定义完整但**没有任何工具或 Agent 代码调用它们**
- [src/utils/retry.py](src/utils/retry.py) — `make_retry_decorator()` 基于 tenacity 的策略封装**从未被应用到任何函数**，实际重试逻辑在 `AgentLoop.run()` 中手动实现

#### J. AnalyzeScreenTool 的 Vision 检测逻辑脆弱

**文件**: [src/tools/ai/vision.py:37-39](src/tools/ai/vision.py#L37-L39)

```python
provider = getattr(llm, "_client", None)
is_anthropic = hasattr(provider, "messages") and not hasattr(provider, "chat")
```

通过访问私有属性 `_client` 来判断是否为 Anthropic 客户端，依赖适配器内部实现细节。如果适配器结构改变，判断逻辑会静默失败（默认为非 Anthropic 路径），导致图像分析实际不走视觉模型。

### 4.4 潜在隐患

#### K. 浏览器单例状态泄漏

**文件**: [src/tools/browser/navigate.py:13-34](src/tools/browser/navigate.py#L13-L34)

`_get_page()` 使用模块级属性存储浏览器实例，问题：
- 多任务共享同一浏览器实例，没有隔离
- 如果浏览器意外崩溃，`is_closed()` 可能抛出异常
- 没有超时或重连机制
- 浏览器关闭后，所有后续浏览器操作都会失败直到进程重启

#### L. 配置缓存导致热更新无效

**文件**: [src/utils/config.py:108](src/utils/config.py#L108)

`load_config()` 使用 `@lru_cache` 装饰，首次调用后 config 被永久缓存。用户在 UI 设置页修改 provider/model 后，`load_config()` 仍返回旧值。虽然 `llm_factory` 会从 `user_settings` 读取 provider，但其他配置项（如 `agent.max_steps`）的热更新会失效。

#### M. Gradio UI 线程阻塞

**文件**: [src/ui/gradio_app.py:453](src/ui/gradio_app.py#L453)

```python
bridge._ready.wait(timeout=30)
```

`on_run` generator 在 Gradio 事件线程中同步等待 `_ready` 事件（最多 30 秒）。如果 LLM 客户端初始化耗时较长（网络问题），UI 在此期间完全无响应。

#### N. Gradio 事件循环中的 nameerror 风险

**文件**: [src/ui/gradio_app.py:511-517](src/ui/gradio_app.py#L511-L517)

```python
yield (
    _build_html(all_events),
    None,
    console_text if "console_text" in dir() else "",
    gr.HTML(status if "status" in dir() else "<div class='footer'>已结束</div>"),
)
```

使用 `"console_text" in dir()` 检查变量是否存在是一种反模式。如果 while 循环从未执行（例如 bridge 线程瞬间退出），`console_text` 和 `status` 变量不会被定义，依赖 `dir()` 检查来兜底。

#### O. ChromaDB 初始化无声失败

**文件**: [src/agent/memory.py:94-96](src/agent/memory.py#L94-L96)

```python
except Exception:
    self._client = None
    self._collection = None
```

ChromaDB 初始化失败时完全静默，不记录任何警告。长期记忆功能失效后，用户和开发者都无法感知。

#### P. Gradio Tab 语法问题

**文件**: [src/ui/gradio_app.py:263](src/ui/gradio_app.py#L263)

```python
with gr.Tab("任务") as task_tab, gr.Tab("设置") as settings_tab:
    pass
```

这种方式在较旧版本的 Gradio 中可能不支持。`gr.Tab()` 作为上下文管理器时，通常需要用嵌套的 `with` 语句：

```python
with gr.Tab("任务"):
    task_tab = ...
with gr.Tab("设置"):
    settings_tab = ...
```

具体行为取决于 Gradio 版本。

#### Q. CloseAppTool 依赖未声明的依赖

**文件**: [src/tools/desktop/app_control.py:115](src/tools/desktop/app_control.py#L115)

`CloseAppTool.execute()` 使用 `import pygetwindow as gw`，但 `pygetwindow` 不在 `pyproject.toml` 的依赖列表中（它是 `pyautogui` 的依赖）。如果环境中恰好没有，工具会在运行时失败。

#### R. ExportPDFTool 使用硬编码的魔数

**文件**: [src/tools/desktop/wps_com.py:286](src/tools/desktop/wps_com.py#L286)

```python
doc.ExportAsFixedFormat(filepath, 17)  # 17 = wdExportFormatPDF
```

使用硬编码数字 `17` 代替 `win32com.client.constants.wdExportFormatPDF`。如果 WPS 的常量值不同（WPS 的 COM 常量可能与 Word 不完全一致），导出会失败。

### 4.5 测试与文档

#### S. 测试目录为空

`tests/` 目录下创建了 `harness/`、`test_agent/`、`test_tools/`、`test_capability/` 四个子目录，但**没有任何测试文件**。测试框架、fixture、mock 均未实现。

#### T. 空目录

- `scripts/` — 设计文档中提到的 `setup.ps1`、`run_zhihu.py` 等脚本不存在
- `docs/` — 完全为空

## 六、总结

Desktop Agent 拥有清晰的架构设计和完整的代码骨架（约 18 个源文件，2000+ 行代码），三层架构（Brain/Hands/Eyes）的设计理念合理，LLM 适配器模式和事件驱动 UI 架构体现了良好的工程思维。

但项目目前处于**原型/骨架阶段**，距离真正可运行还有关键差距：

1. **桌面工具在 UI 模式下因 COM 线程问题无法运行**
2. **感知层与 Agent 循环的数据流未打通**，验证系统形同虚设
3. **微信和知乎的快捷模式没有对应的工具支撑**
4. **缺乏任何测试覆盖**，回归风险高

建议优先修复 COM 初始化问题（P0），然后打通感知-验证链路（P1），再补齐微信/知乎工具（P1），最后完善测试体系（P3）。
